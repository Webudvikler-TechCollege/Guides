GitHub Side notes
For daily use

Step 9: Add items to the backlog (see detailed version here)

After the design sprint your group will have some items in the backlog but during the project you will probably find more things that need to be added to the backlog. You could add these tasks during or after the daily standup or at any other time, just make sure your whole group knows about it

Step 10: Drag issues to the "Ready" column

Before each "sprint" the group decides what tasks you are going to work on for that "sprint". When you have decided what tasks you are going to work on, drag them to the "Ready" column.

Step 11: Convert the item to an issue

If the item is something that needs to be coded, you need to convert it to an issue so that it can be marked in a pull request.

Step 12: Assign issue to a repository

Some projects might have two repositories (one for the back-end and another for the front-end) and in that case you assign the issue to the correct repository. If you only have one repository you just assign all the issues to that repository.

Step 13: Assign issues to developers

When you create the issue you might already know who is going to work on it, if that is the case you can assign the issue to that person. If you don't know yet who is going to work on it you can skip this step

Step 14: Drag issues to the "In-Progress" column (see detailed version here)

When you have decided what issue you want to work on you drag it to the "In-Progress" column.

Step 15: Create a branch locally

  1. Make sure you are on the develop branch
  2. Pull from develop branch
  3. Make, and checkout your own branch named issue#nr (where nr is the number of the issue you are trying to resolve)

Step 16: Work on the issue

change and add code to the project until you have solved the issue

Step 17: push your code to your new branch

Write the following in your terminal:

  1. git add .
  2. git commit -m "issue#nr - some text (where nr is the number of your issue and "some text" is a description of what you did)
  3. git push origin issue#nr (remember to change the nr to the number of your issue)

Step 18: Create a pull request

Go to GitHub and create a pull request from your branch into the devlop branch

Keep in mind

Remember to check on pending pull requests regularly and give helpful code reviews for them. To get started with code reviews you can try to compy-paste the code from the pull request into chatGPT and ask it to help you make a code review
